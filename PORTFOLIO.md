# Job Search Bot
**채용 공고 자동 수집 · 검색 · 알림 시스템**

`Python` `Playwright` `PostgreSQL` `Discord.py` `EXAONE 3.5 7.8B` `Ollama` `Cron`

---

## Overview

잡코리아에서 매일 신규 채용 공고를 자동으로 수집하고, Discord를 통해 자연어로 검색하거나 관심 조건을 구독하면 맞춤 공고를 DM으로 자동 알림하는 end-to-end 데이터 파이프라인을 설계·구현했다.

- 크롤링 → 정제 → 저장 → 검색 · 알림까지 전 과정 직접 구현
- testset 100건 기반 검색 품질 측정 후, LLM 3가지 방안으로 수치 개선 증명

---

## Architecture & Flow

  Cron job                      # 매일 06:00 크롤링 파이프라인 자동 실행                                    
  → *Scraper                    # Playwright headless 브라우저로 잡코리아 동적 페이지 크롤링                                                                                        
  → *JobPreprocessor            # 경력·연봉·학력·고용형태·지역을 숫자 코드로 정규화.
  → EXAONE via Ollama           # 공고명과 기존 태그를 입력으로 직무·기술 시맨틱 태그 자동 생성(데이터 보강)
                                사용자 자연어 쿼리로부터 검색 태그 추출 및 보강, 결과 후보 재순위 및 선정
  → PostgreSQL                  # recruits · tags · companies · regions 정규화 5개 테이블
  → *Market Snapshot            # 유효 공고 기준 인기 태그·평균 연봉·지역·경력 분포를 집계
  → *Discord Bot                # 자연어 쿼리 정규식 파싱 → SQL 검색 → LLM 재순위 → 결과 반환,
                                사용자 구독 태그 신규 공고 매칭 후 DM 발송 (24h 주기)

## Trouble Shooting

### 1. FAISS 벡터 검색 제거 → SQL 전환

**Problem**
KR-SBERT + FAISS 기반 벡터 검색을 도입했으나 검색 품질이 기대에 미치지 못함.

**Cause**
세 가지 구조적 한계가 있었음.
- 임베딩 대상이 공고명 하나뿐이라 지역·경력·연봉 조건이 벡터 유사도에 반영되지 않음
- 잡코리아 공고 설명이 자연어 문단이 아닌 `"Python, Django, REST API"` 형태의 키워드 목록이라 시맨틱 임베딩 효과가 낮음
- FAISS 인덱스(~430MB) + KR-SBERT 모델(~300MB) = 730MB 상시 메모리 점유

**Solution**
FAISS를 제거하고 PostgreSQL SQL 기반 검색으로 전환. 지역·경력·연봉·고용형태는 정형 컬럼 WHERE 절로, 직무·기술 키워드는 공고명·태그 ILIKE 검색으로 처리. ~730MB 메모리 회수.

---

### 2. 검색 품질 정량 평가 — testset 구축

**Problem**
검색 기능을 수정할 때마다 품질이 나아졌는지 주관적으로만 판단 가능. 개선 효과를 수치로 증명할 방법이 없었음.

**Cause**
정량 평가를 위한 testset과 지표(metric)가 없었음. 어떤 쿼리에 어떤 공고가 기대 결과인지 정의된 기준이 없는 상태에서 개발 진행.

**Solution**
LLM에게 "이 공고를 찾을 법한 검색 쿼리를 만들어줘" 방식으로 testset 100건을 역방향 생성하고, 3가지 지표로 기준선을 측정.

- **Hit@K**: 상위 K건 결과 안에 기대 공고가 포함되는 비율
- **MRR** (Mean Reciprocal Rank): 기대 공고가 몇 번째에 등장하는지 평균 역순위 — 높을수록 상위 노출
- **Zero-result rate**: 검색 결과가 0건인 쿼리 비율

방안별로도 전용 평가 스크립트(구독 매칭 Before/After, Precision@K, Recall@K)를 각각 구축해 각 개선의 효과를 독립적으로 측정할 수 있는 평가 체계를 마련.

기준선: Hit@5=52%, Hit@10=58%, MRR=0.394, Zero-result=3%

---

### 3. 검색 품질 향상 — LLM 3가지 방안 도입과 평가

**Problem**
구축한 testset 기반 기준선 측정 결과 Hit@5=52%, Hit@10=58%, Zero-result=3%. 동의어·관련 기술 스택으로 검색 시 관련 공고가 누락되고, 매칭된 후보 중 관련도 정렬이 없어 상위 노출 품질이 낮음.

**Cause**
- 키워드 ILIKE 매칭만으로는 `"프론트 엔지니어"` → `"React"`, `"Vue"` 공고 미커버 (Recall 한계)
- 후보 공고를 관련도 순으로 정렬하는 로직 없음 (Precision 한계)
- 공고명에 없는 동의어로 검색 시 색인 자체가 부재

**Solution**
3가지 방안을 순차 도입하고 방안별 평가 스크립트로 효과를 측정.

- **방안 1 (재순위)**: 후보 공고를 LLM 관련도 평가로 재정렬 → Precision@10 45.0% → 64.0% (+19.0%p)
- **방안 2 (시맨틱 태깅)**: 크롤링 시 공고명 기반으로 직무·기술 태그 자동 생성 → Recall@10 37.8% → 88.9% (+51.1%p)

| 방안 | 지표 | Before | After |
|---|---|---|---|
| 재순위 | Precision@10 | 45.0% | 64.0% (+19.0%p) |
| 시맨틱 태깅 | Recall@10 (동의어 쿼리) | 37.8% | 88.9% (+51.1%p) |

---

### 4. TREC-style 평가 프레임워크 구축 — 검색 품질 구조적 한계 진단

**Problem**
방안 1·2 도입 후 수치가 개선됐지만, 평가 설계(역방향 쿼리 생성 방식)가 실제 사용자 쿼리를 대표하지 못할 수 있었음. 시맨틱 태그 기반 검색이 진짜 도움이 되는지, 아니면 평가 방법이 편향된 것인지 확신이 없었음.

**Approach**
정보 검색 표준 평가 방법론인 **TREC-style pooling**을 직접 설계해 재평가.

- 쿼리 유형 4가지(키워드 직접형·시맨틱형·모호형·엣지케이스)로 43개 쿼리 수작업 설계
- 5가지 모드(baseline / +tags / +rerank / +expanded / +adaptive)를 동시 실행해 후보 풀 수집
- 룰 기반 판정자(직무·지역·고용형태·경력·연봉 기준 0-3점)로 관련도 판정
- NDCG@K, Precision@K, Hit@K로 모드별 성능 비교

**Findings**

| 모드 | NDCG@10 | vs baseline |
|------|---------|-------------|
| baseline (SQL AND) | 0.753 | — |
| +rerank | **0.771** | **+0.017** |
| +tags (태그 AND 매칭) | 0.657 | -0.096 |
| +tags+rerank | 0.659 | -0.094 |
| +expanded (항상 LLM 확장) | 0.527 | -0.137 |
| +adaptive (결과 부족 시만 LLM 확장) | 0.667 | **+0.009** |

- **태그 기반 검색이 오히려 성능을 낮춤**: 멀티토큰 AND 매칭이 과적용되어 결과 0건 케이스 증가
- **+rerank만이 일관된 개선**: NDCG@10 +0.017. 단, B-type(시맨틱 쿼리) 제외 시 +0.003으로 미미
- **+adaptive**: baseline 초과 달성한 유일한 LLM 확장 모드. 역시 시맨틱 쿼리 제외 시 효과 없음

**Root Cause 진단**

평가 과정에서 판정 방법론 자체의 편향도 발견했음.

- 룰 기반 판정자가 `announcement_name` 기반 키워드 매칭으로 관련도를 판정
- baseline 검색도 동일한 `announcement_name` 키워드 매칭 → 판정자와 baseline이 같은 신호를 봄
- 태그·LLM 기반 검색이 가져오는 "의미적으로 관련된 공고"는 제목이 달라 판정자가 낮은 점수를 줄 수 있음

근본 원인은 **데이터 품질**: 공고 제목 + 잡코리아 제공 태그만 보유하고 있어, 공고 상세 설명 없이는 어떤 검색 방식을 도입해도 의미 있는 개선의 한계가 명확함. 상세 페이지는 대부분 PDF 제공으로 대량 수집이 어렵고, 공개 API(사람인·워크넷)도 상세 텍스트는 미제공.

**Decision**

추가 엔지니어링보다 **방향 전환이 합리적**이라는 결론. 현재 검색 구조(+adaptive + +rerank)를 유지하되, 데이터 소스 다변화를 다음 프로젝트 주제로 설정.

### 5. LLM 오버엔지니어링 — 검색 필터 추출

**Problem**
LLM(Gemma3:4b)으로 쿼리 필터 추출 시 응답 수 초 지연, JSON 파싱 실패, Ollama 상시 의존 문제 발생.

**Cause**
LLM이 추출하는 조건(지역·고용형태·경력·연봉)을 분석하니 모두 유한 목록 매칭과 정규식으로 처리 가능한 정형 패턴이었음. 키워드 확장·재순위 등 의미 이해가 필요한 작업과 달리, 필터 추출은 LLM이 필요한 영역이 아니었음.

**Solution**
LLM 제거 후 정규식 기반 파싱으로 교체. 응답속도 수 초 → 밀리초, Ollama·LangChain 의존성 완전 제거,
  7개 필터 결정론적 동작.
